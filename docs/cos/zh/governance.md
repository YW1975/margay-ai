# 治理:dual-loop 与拍板

> 给老板用、且会动到"对人催办、对外承诺"的 AI,**治理是敢被放进管理层的前提**。这是诸葛喵与"放飞自动化"的根本区别。

## 三条铁律
1. **默认 dry-run** —— 一切对外动作(催办/发消息/写表)默认**只起草不发**,需老板显式同意才真发。
2. **经治理门** —— 任一外发前过 dual-loop:**propose → critic → 裁决**。
3. **fail-closed** —— 出错或没批准,绝不外发。

## dual-loop 治理门
```
提议(propose) → 批判(critic) → 裁决(decide)
```
裁决三态:
- **APPROVE** —— 低风险、critic 通过 → 可发(仍 dry-run 默认,除非显式 live)
- **NEEDS_APPROVAL** —— **L4 / owner-only / 高风险(stakes)→ 待老板拍板,绝不自动发**
- **REJECT** —— 绕过 critic(bypass)、越权 → 拒绝执行

`cos_cli.py gate --text ... --target ... --level 1-4` 即调此门。

## 为什么这是护城河
竞品的 agent 要么在自己权限内自动做(飞书/钉钉),要么是个人桌面执行(WorkBuddy)—— **没有"管理级治理 + 老板始终在控制位"**。诸葛喵把治理做成第一公民,所以老板敢把"对人催办、对外承诺"这类高风险动作托付给它:
- 老板永远是**最终决策者**;诸葛喵负责把活干到"待发"那一步。
- 这让硅基智能第一次能安全地进入**管理决策层**,而不只是做做信息整理。

## 与升级级别配合
催办带 escalation level(L1 提醒 → L2 求助/协调 → L3 升级 → L4 待老板拍板)。级别越高、风险越大,越往 NEEDS_APPROVAL 走 —— 把"该自动的自动、该请示的请示"分得清清楚楚。
